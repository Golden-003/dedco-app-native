01 — Vision

Vision & positionnement

L'application mobile Dedcco n'est pas une déclinaison du site web. C'est un produit mobile premium conçu pour devenir la référence de l'aménagement intérieur au Bénin. Lorsqu'une personne pense « je veux décorer », « je cherche un artisan », « je veux refaire mon salon », le premier réflexe doit être : j'ouvre Dedco.

Positionnement

Le web est la plateforme complète — utilisateurs, artisans, designers, maisons déco, administration, gestion. Les prestataires gèrent leurs actions depuis le web. Le mobile est uniquement pensé pour le client. Les professionnels utilisent principalement le web.

Cette séparation est essentielle : l'app mobile n'a pas besoin de gérer des dashboards admin, des statistiques business, ou des workflows artisan complexes. Elle se concentre sur l'expérience client — découverte, inspiration, achat, suivi de projet, coordination.

Les 3 piliers

PilierDescriptionApplication
ConversionChaque écran conduit naturellement vers une action qui crée de la valeurEnregistrer une inspiration, contacter un designer, créer un projet, ajouter au panier, commander
Anti-contournementChaque interaction maintient la relation commerciale dans DedcoPas de téléphone, WhatsApp, Facebook, email, paiement externe. L'utilisateur ne ressent jamais la contrainte.
Empreinte mentaleL'application doit donner envie d'être ouverte, même sans intention d'achatComme Pinterest, Airbnb, Instagram. Dedco doit devenir une habitude.
Règle d'or. L'utilisateur ne doit jamais avoir l'impression de naviguer. Il doit avoir l'impression de découvrir. Chaque écran doit être vivant, immersif, fluide, émotionnel, premium.

Architecture mentale

L'utilisateur ne doit jamais penser « je cherche un meuble ». Il doit penser « je cherche une idée ». Puis : « Dedco m'aide à transformer cette idée en réalité ». C'est cette différence qui distingue une marketplace d'un véritable produit de référence.

03 — Accueil

Écran Accueil

Le cœur de l'application. L'objectif est de donner immédiatement envie d'aménager son intérieur.

Structure

  • Hero immersif — image plein écran, plein cadre, avec overlay gradient. Le visuel change à chaque ouverture pour renouveler l'expérience.
  • Élément de personnalisation — un carrousel horizontal d'inspirations basées sur le style préféré de l'utilisateur (si connecté) ou des tendances générales.
  • Coups de cœur — 4-6 produits vedettes en cards horizontales scrollables (swipe).
  • Scènes interactives — 1-2 scènes « shop the look » avec hotspots cliquables.
  • Artisans à découvrir — 2-3 profils d'artisans mis en avant.
  • CTA brief — section « Deux façons de créer » (brief artisan + brief designer).

Élément de renouvellement

L'accueil doit intégrer un élément fort de personnalisation ou de découverte qui renouvelle l'expérience à chaque ouverture, afin que le flux paraisse vivant et non figé. Concrètement :

  • Rotation du hero — 5-10 visuels différents, un aléatoire à chaque ouverture
  • Sélection personnalisée — si l'utilisateur a des favoris ou un style préféré, les coups de cœur s'adaptent
  • Section « Nouveautés » — produits ajoutés dans les 7 derniers jours
  • Tendance de la semaine — une scène ou collection mise en avant éditorialement
04 — Inspirations

Écran Inspirations

Le moteur émotionnel de l'app. Chaque inspiration doit pouvoir conduire à un produit, un designer, un artisan ou un projet.

Structure

  • Masonry grid — grille Pinterest-style, 2 colonnes, scroll infini
  • Filtres par style — Afro-contemporain, Tropical luxe, Zen minimaliste, Éclectique, Biophilique, Traditionnel revisité
  • Tap sur une image — ouvre l'inspiration en plein écran avec :
    • Produits taggés (shop the look)
    • Designer ou artisan associé
    • Bouton « Créer un projet similaire »
    • Bouton « Sauvegarder » (ajoute aux favoris)
  • Swipe horizontal — naviguer entre inspirations en plein écran
  • Pinch to zoom — zoomer sur une image en plein écran
Conversion. Chaque inspiration doit avoir au moins un point de sortie : un produit, un designer, ou un brief. Aucune inspiration « morte » sans action possible.
05 — Marketplace

Marketplace & fiche produit

La marketplace est optimisée pour la recherche rapide mobile. La fiche produit est très immersive.

Marketplace

  • Recherche — barre de recherche sticky en haut, autocomplétion
  • Filtres en bottom sheet — catégorie, prix, niveau artisan, note, zone géographique. Style Airbnb.
  • Grid 2 colonnes — cards produit avec image, nom, prix, badge artisan vérifié
  • Tri — pertinence, prix croissant, prix décroissant, nouveautés, mieux notés
  • Collections — sections éditoriales mises en avant (ex: « Made in Cotonou », « Bois d'iroko »)

Fiche produit

  • Galerie plein écran — swipe horizontal entre photos, pinch to zoom, possibilité de vidéo
  • Bottom sheet produit — glisse depuis le bas : nom, prix, description, artisan, options (couleur, dimensions), ajout au panier
  • Produits similaires — carrousel horizontal en bas de page
  • Produits associés — « Complétez votre espace » (produits qui vont avec)
  • Avis — notes et commentaires clients, avec photos si disponibles
  • Profil artisan — lien vers le profil de l'artisan (consultation uniquement)
Geste clé. Le tap sur l'image produit ouvre la galerie plein écran. Le bottom sheet produit reste accessible en glissant depuis le bas. L'utilisateur peut voir l'image en grand ET les infos produit simultanément.

07 — Panier

Panier & checkout

Optimisé Mobile Money. Peu d'étapes, très rassurant. Le checkout se fait en bottom sheet.

Panier

  • Liste des articles avec image, nom, prix, quantité (stepper + / -)
  • Suppression par swipe gauche (gesture native)
  • Sous-total, frais de livraison, total en .font-numeric
  • Bouton « Commander » sticky en bas

Anti-contournement. Le paiement ne passe jamais par un canal externe. Mobile Money est intégré dans Dedco. L'utilisateur ne quitte jamais l'app. Aucun échange de numéro, aucun WhatsApp pour « finaliser ».
08 — Chat

Chat de coordination

Principe fondamental. Le chat n'est pas une messagerie. C'est un module de coordination lié aux projets. Pas de chat libre. Pas de conversation sans contexte projet.

Quand le chat apparaît

  • L'utilisateur crée un brief → un thread de chat est créé automatiquement quand un artisan/designer répond
  • L'utilisateur passe une commande → un thread de chat est créé avec l'artisan pour le suivi
  • L'icône Messages dans le header n'apparaît que si l'utilisateur a au moins un thread actif
  • L'utilisateur ne peut pas « démarrer une conversation » — il n'y a pas de bouton « nouveau message »

Autorisé dans le chat

ActionContexte
Poser des questionsSur le projet, les matériaux, le délai
Envoyer des photosPhotos de suivi, d'inspiration, de validation
Suivre un projetÉtapes, avancement, photos de fabrication
Confirmer une étapeValidation de modification, acceptation de livraison
Échanger des documentsDevis, factures, plans — liés au projet

Interdit dans le chat

InterdictionMécanisme
TéléphoneDétection de numéros dans les messages → masqués automatiquement
EmailDétection d'adresses email → masquées
Réseaux sociauxBlocage des liens WhatsApp, Facebook, Instagram
Liens externesSeuls les liens vers des produits/profil Dedco sont autorisés
Paiement externeAucun échange d'argent en dehors de Dedco
Le chat doit être relié aux objets métier. Chaque conversation est attachée à une commande, un projet, ou une prestation. Pas à une conversation libre. L'utilisateur accède au chat depuis le suivi de projet, pas depuis un bouton « messages » générique.

UX du chat

  • Liste des conversations — accessible depuis l'icône Messages (header) ou depuis le suivi de projet
  • Chaque conversation affiche le projet associé en header (titre, statut, progression)
  • Messages — bulles classiques, photos en galerie inline, documents en pièces jointes
  • Actions contextuelles — valider une modification, confirmer livraison, demander révision
  • Notifications — push notifications pour nouveaux messages, badge dans le header
09 — Compte

Écran Compte

Centre de gestion de l'utilisateur. Accès aux commandes, projets, paiements, paramètres et threads de chat.

Structure

  • Profil — avatar, nom, email, rôle (client)
  • Mes commandes — liste avec statut, suivi, accès au chat de coordination
  • Mes projets — briefs envoyés, propositions reçues, projets en cours, projets livrés
  • Paiements — historique des transactions, factures
  • Adresses — adresses de livraison sauvegardées
  • Paramètres — notifications, langue, confidentialité, déconnexion
  • Messages — accès aux threads de chat par projet actif

11 — DS

Conformité Design System

Le Design System est la source unique. Aucun composant inventé. Le mobile est une adaptation fidèle, jamais une réinterprétation.

Respect absolu

  • Palette officielle — Amber #BF793B, Terracotta #A6442E, Forest #548C45. Pas de 4e couleur.
  • Typographie — Quache (display, mots) + Plus Jakarta Sans (corps, chiffres). Pattern unicode-range pour les chiffres.
  • Espacements — échelle Tailwind (4, 8, 12, 16, 20, 24, 32, 40, 56, 64)
  • Rayons — sm 4px, md 6px, lg 10px, xl 14px, 2xl 20px
  • Ombres — xs → xl + shadow-amber, teintées chaudes rgba(30,24,19,X)
  • Cards — .dedco-card avec hover translateY(-2px)
  • Boutons — .dedco-btn-* + .dedco-btn-icon pour boutons icônes
  • Icônes — lucide-react, stroke 2px, linecap round, fill none, currentColor
  • États — hover, active, disabled, focus-visible (outline 3px amber)
  • Animations — fade-up, scale-in, shimmer, pulse, stagger-1 à 6
Règle. Tout composant mobile doit pouvoir être rattaché à un token ou une classe du Design System. Si un composant n'existe pas dans le DS, il faut d'abord l'ajouter au DS, puis l'utiliser dans le mobile.
12 — Anti-contournement

Anti-contournement — règles

Chaque interaction doit maintenir la relation commerciale dans Dedco. Le produit ne doit jamais faciliter l'échange hors plateforme.

InterditMécanisme de protection
Échange de téléphoneDétection automatique dans le chat → masquage. Pas de champ téléphone visible sur les profils publics.
WhatsAppBlocage des liens WhatsApp dans le chat. Pas de bouton WhatsApp sur les profils.
Facebook / InstagramPas de liens sociaux. Pas de boutons de partage vers des réseaux externes.
EmailDétection d'adresses email dans le chat → masquage. Communication uniquement via le chat Dedco.
Paiement externeMobile Money intégré. Pas de virement bancaire direct, pas de cash à la livraison sans traçabilité.
L'utilisateur ne doit jamais ressentir cette protection comme une contrainte. Il doit simplement avoir tout ce dont il a besoin dans Dedco. Si l'expérience est suffisamment complète et fluide, l'utilisateur n'aura jamais envie de contourner.
13 — Stack

Stack technique recommandée

TechnologieRôleRaison
React Native + ExpoFramework mobile cross-platformPartage du code avec le web (même React), déploiement iOS + Android, accès natif (caméra, notifications push, gestures)
Expo RouterNavigation mobileÉquivalent App Router pour mobile, file-based routing, tabs + stacks
Zustand (persist)State managementDéjà utilisé sur le web, partage possible des stores (cart, favorites, user)
Tailwind (NativeWind)StylingMême syntaxe que le web, mêmes tokens, mêmes classes .dedco-*
lucide-react-nativeIcônesMêmes icônes que le web, cohérence visuelle totale
Reanimated 3Animations gestuellesSwipe, bottom sheets, micro-animations 60fps
API Dedcco (partagée)BackendMêmes routes API que le web (/api/products, /api/orders, /api/auth)

Partage de code

  • Design tokens — les mêmes variables CSS/JS (palette, radii, shadows, fonts)
  • Types — les mêmes types TypeScript (Product, Artisan, Brief, etc.)
  • Stores — Zustand stores partagés (cart, favorites, auth)
  • API client — mêmes routes API, mêmes payloads
  • Logique métier — moteur brief-artisan partagé (transitions, permissions, engine)
Architecture. Un monorepo (Turborepo ou Nx) avec 3 packages : web (Next.js), mobile (Expo), shared (types, tokens, stores, engine). Le code partagé évite la duplication et garantit la cohérence.
